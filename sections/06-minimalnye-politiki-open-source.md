## Минимальные политики Open Source — подробное описание

Цель политик — обеспечить минимум качества и безопасности, при котором OSS-слой не превращается в кладбище репозиториев.

### 1) Лицензия по умолчанию
**Дефолт:**
- код: **Apache-2.0** (патентная оговорка, удобна для B2B-интеграций);
- документация/контент: **CC BY 4.0**.

**Правила:**
- лицензия должна быть в репозитории (LICENSE) и отражена в `README`.
- если нужен другой режим (MIT/BSL/dual-license/закрытая часть) — это “техническое решение” и фиксируется:
  - кто owner,
  - какие части под какой лицензией,
  - почему это нужно,
  - как будет сопровождаться.

### 2) Security disclosure
**Канал:**
- `security@prizm.community` (или GitHub Security Advisories).

**Запрет:**
- не публиковать 0-day публично до фикса.

**Шаблон репорта:**
- затронутые версии,
- вектор атаки,
- влияние (CIA),
- PoC (если есть),
- рекомендации по фиксу (если есть).

**SLA security:**
- ack: до 72 часов,
- план/оценка: до 7 дней,
- фикc+релиз:
  - high/critical: до 30 дней,
  - medium/low: до 90 дней (если нет активной эксплуатации).

**Disclosure:**
- advisories + release notes,
- CVE опционально (при необходимости и наличии ресурсов).

### 3) SLA triage / review (для “active” репозиториев)
**Triage (issues):**
- ack: до 72 часов;
- первичная классификация (bug/feature/security/docs), приоритет, назначение: до 7 дней;
- решение “accepted / needs info / wontfix”: до 14 дней.

**Review (PR):**
- первая реакция: до 3 рабочих дней;
- первое содержательное ревью: до 7 дней;
- после правок автора: повторное ревью до 7 дней.

**Эскалация при нарушении SLA:**
- делегировать ревью/triage другому maintainer’у,
- либо пометить репозиторий как `paused/stale`,
- либо инициировать набор сопровождающих через гильдию.

### 4) Требования к релизам
**SemVer** обязателен.

Минимальный “release gate”:
- зеленый CI (сборка/тесты/линтер),
- базовый dependency scan / SAST (если инфраструктура есть),
- changelog + release notes,
- описание breaking changes и миграций.

Политика совместимости:
- deprecation минимум 1 minor-версия до удаления/ломающего изменения,
- major-изменения сопровождаются migration guide.

Процесс релиза:
- есть ответственный (maintainer или `release shepherd`),
- релиз происходит из стабильной ветки,
- теги/релизы подписываются, если доступно,
- по возможности делаются воспроизводимые сборки.
